## Why this exists

User-facing strings in source rot fast (inconsistent casing, missed translations, unintentional brand drift). A single catalog source of truth + per-platform codegen + a strict-zero ratchet on bare literals gives you: localizable surfaces from day 1, a single place to review wording, and structural defense against new bare-literal regressions. Multi-locale BCP-47 with a documented resolver chain (`Member.locale ‚Üí Household.locale ‚Üí device ‚Üí en-US`) handles per-user overrides cleanly.

## What you must satisfy

- `shared/strings/catalogs/<locale>.json` ‚Äî source of truth per locale (start with `en-US`; add others as the product needs them).
- A `t()` helper per platform, consuming a per-platform generated file:
  - Web TS: `src/lib/strings/strings.generated.ts`
  - iOS Swift: `packages/<Core>UI/Sources/<Core>UI/Generated/LocalizedStrings.generated.swift`
  - CF-side TS: `shared-cf-utils/src/generated/strings.generated.ts`
- Codegen runner `scripts/gen-strings.mjs` emits all three from the catalog; `npm run strings:gen` regenerates, `npm run strings:check` verifies in sync.
- A locale resolver:
  - Member.locale (per-user override)
  - Household.locale (shared default)
  - Device/browser locale (auto-detect)
  - Fallback to `en-US`
- Strict-zero ratchet `no-bare-user-facing-string-in-features` defends `src/{features,pages,components}/` (web) AND `apple/<App>/<App>/**/*.swift` + `packages/<Core>UI/**/*.swift` (iOS).
- BRAND_STRINGS allowlist (project name, integration names like `Google`/`Apple`/`Firebase`) bypass the ratchet.
- All onCall CF wire codes route through `errors.<code>` catalog keys via `cfErrorHandler.ts` ‚Üí `t()`.
